## Storm Probability Fix (Task 3)

| Method | Storm chance (72h) |
|---|---|
| **Before** (mean-trajectory): `np.mean(kp_weighted >= 5)` | **0.0%** (structurally suppressed) |
| **After** (500-draw ensemble): `predict_scenario_kp_ensemble` | **27.6%** (near-solar-max, current quiet) |

The old `storm_chance_percent = 0.0` for every quiet run was a concrete bug: the
weighted-mean Kp trajectory never exceeded 5.0 during quiet conditions, so the
probability was always exactly zero.  The ensemble method correctly reflects the
non-zero probability mass in the Active scenario tails.  During solar maximum
(current cycle 25 peak), a 72h window carries ~10–30% base rate for G1+ activity
depending on conditions — 27.6% on a quiet day is physically reasonable.

The three new features (`equinox_term`, `vbs`, `kp_lag_27d`) show near-zero
permutation importance on the 2015–2022 test window.  This is an honest result,
not an error.  `kp_lag_3h` already encodes most of the autoregressive signal,
leaving little marginal information for the other lag features.  `equinox_term`
and `vbs` may become more important on longer training windows or during periods
with strong semi-annual modulation (e.g., CIR-dominated solar minima).  Keeping
them in the feature set costs nothing and ensures the model can exploit them when
the signal is present.

## Summary

The model beats the persistence baseline (skill ratio 0.699, ΔR²=+0.277) across
all 8 training years.  Reliability is excellent.  Storm detection (POD=0.329,
CSI=0.302) is reasonable for a regression model without dedicated extreme-event
weighting.  If storm detection is the primary use case, the next step would be a
calibrated probability threshold or a separate binary classifier trained
specifically on the G1+ class.
