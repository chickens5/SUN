What this page highlights

Solar wind ingestion from NOAA SWPC and the OMNI fallback path.
Physics-led feature engineering with Newell coupling and time-aware splitting.
Held-out evaluation, storm-vs-quiet regime analysis, and operational forecast cells.
01 / Ingestion

Live solar wind streams

The notebook pulls NOAA plasma, magnetic field, and planetary Kp feeds, then falls back to a synthetic but physically plausible dataset when the network is unavailable.

02 / Modeling

Chronological training

Multi-year OMNI data provides a stronger time-series split and a more trustworthy error profile than a tiny random slice of one week.

03 / Forecasting

Scenario ensemble

Quiet, moderate, and active futures are projected for the next 72 hours so the notebook can communicate risk instead of just emitting a single point estimate.

Implementation

How this static version works

Step 01

Keep the notebook as source

The production notebook remains the canonical artifact. This page does not execute it; it explains the pipeline and links back to the .ipynb file.

Step 02

Export or rewrite to HTML

For GitHub Pages, the notebook can be exported to static HTML with nbconvert or mirrored into a hand-authored HTML page like this one.

Step 03

Publish the generated files

GitHub Pages only needs static files. Point the Pages source at the branch or folder that contains this HTML and any exported assets.

Publish

GitHub Pages checklist

Build step No backend, no server, and no notebook runtime. Just HTML, CSS, and optional images.
Site source Use the repository root, the docs/ folder, or a dedicated gh-pages branch.
Notebook link Keep the .ipynb file in the repo so people can inspect the original analysis.
Asset paths Use relative URLs only so the site works from the GitHub Pages subpath.

Minimal publish recipe

A fully static deployment can be published with one exported page and one optional notebook link. The example below is the exact shape this repo is using now.

github pages 
1. Keep GGSP_v7.0.html as the public landing page.
2. Keep GGSP_v7.0_frontend.html as the static implementation note.
3. Keep GGSP-7.0_PROD.ipynb in the repo for source inspection.
4. Turn on GitHub Pages for the branch or folder that contains these files.
5. Use only relative links so the site works at https://<user>.github.io/SUN/.
